# 嵌入式名词扫盲

> **原则：每个词挂对场景，读者不用换档。**
> 
> 反面案例：在 SMBus 协议文档里开独立小节写 BOM —— 读者正跟着通信协议走，突然跳到物料兼容性，脑子得换档。正确的做法：BOM 作为附注挂在 SLAVE_ID 下面一句话带过，或者放生产级分类里单独说。

## 场景分类

### 电路级（修板子用）
芯片通讯地址、引脚配置、信号波形——修主板的人拿万用表和示波器会查的词。

| 术语 | 一句话 | 常见场景 |
|------|--------|----------|
| **SLAVE_ID** | SMBus/I2C 从设备地址（7-bit），EC 靠这个知道在跟谁说话 | 充电芯片不响应 → 查 SLAVE_ID 对不对 |
| **GPIO** | EC 的通用输入输出引脚，可配置为输入（读电平）或输出（拉高/拉低） | 查开机信号、复位信号来源 |
| **上拉/下拉** | 电阻把悬空引脚固定到确定电平——上拉=VCC，下拉=GND | SDA/SCL 没波形 → 查上拉电阻 |
| **SDA/SCL** | I2C/SMBus 的两条线——SDA 传数据，SCL 传时钟 | 示波器抓波形确认通信是否正常 |
| **PWM** | 脉宽调制，靠占空比控制电压/转速 | 风扇控制、背光调节 |
| **ADC** | 模数转换，把电压变成数字值 | EC 读电池电压、温度传感器 |

### 协议级（调通讯用）
寄存器、命令字、应答机制——调固件和抓逻辑分析仪时会碰到的词。

| 术语 | 一句话 | 常见场景 |
|------|--------|----------|
| **ACK/NACK** | I2C 应答位——从设备收到数据后回 ACK(0) 表示收到，NACK(1) 表示没收到/不想收了 | 逻辑分析仪看到 NACK → 地址错了或芯片挂了 |
| **寄存器映射** | 芯片内部寄存器地址和功能的对应表 | 查"充电电流设哪个寄存器" |
| **PMBus** | 电源管理总线，SMBus 的超集，定义了标准电源命令字 | 调数字电源芯片的电压/电流/故障读取 |
| **SMBus** | 系统管理总线，I2C 的子集，加了超时和错误恢复 | 笔记本 EC 和充电芯片/电池之间的通信协议 |
| **I2C 超时** | SMBus 特有的 25-35ms 超时机制——从设备太久不响应，主设备主动放弃并复位总线 | EC 读充电芯片超时 → 寄存器读回全 FF → 查 SLAVE_ID 或芯片是否掉电 |
| **SVID** | 串行 VID 协议，Intel CPU 用三条线（CLK/DATA/ALERT#）跟 VCore 稳压器通信，动态调电压 | 修 CPU 供电——示波器夹 SVID 三线，看 EC/CPU 有没有在给 VRM 发电压指令 |
| **PSID** | 平台串行 ID，CPU 上电前用一根数据线拉低/拉高告诉 VRM 该平台要用哪套参数 | 换 CPU 后 VCore 不对 → 查 PSID 电平配置是否匹配新平台 |
| **PROCHOT** | Processor Hot，CPU 过热时拉低的信号——充电芯片也能触发，不一定来自 CPU | 降频/不开机 → 查 PROCHOT 是 CPU 自己拉的还是充电芯片/EC 拉的 |
| **SMBus Alert** | SMBus 设备主动通知主机的机制——设备拉低 ALERT# 线，主机轮询所有设备找谁发出的 | 电池/充电芯片有故障时主动通知 EC，不需要 EC 定期轮询 |

### 生产级（跟工厂说话用）
物料、工艺、生产文件——跟工厂采购和贴片线沟通时的词。

| 术语 | 一句话 | 常见场景 |
|------|--------|----------|
| **BOM** | Bill of Materials，物料清单——一块板子所有元器件的型号、封装、用量 | 工厂采购/报价、确认替代料 |
| **Gerber** | PCB 制造文件，每层的走线图形 + 钻孔数据 | 发给 PCB 厂做板子 |
| **钢网** | 刷锡膏用的不锈钢模板，对应 PCB 焊盘开孔 | SMT 贴片前刷锡膏 |
| **SMT** | 表面贴装技术——机器把元件贴到 PCB 上再过回流焊 | 量产阶段 |
| **回流焊** | 高温炉把锡膏熔化焊住元件 | SMT 的最后一步 |

### 调试工具级
修板子/调固件时手里拿的工具和软件。

| 术语 | 一句话 | 常见场景 |
|------|--------|----------|
| **逻辑分析仪** | 抓数字信号时序的工具，能同时看多路信号 | 抓 SDA/SCL 波形看 I2C 通信 |
| **示波器** | 看模拟波形和电压时序 | 查上电时序、纹波、信号质量 |
| **EC 串口 console** | EC 的调试串口，可以敲命令读寄存器/看日志 | 不开机时看 EC 卡在哪一步 |
| **万用表** | 量电压/电阻/通断 | 第一步——"有没有电？" |

## 反面案例：BOM 为什么不能出现在协议文档里

背景：4 妹在 SMBus 协议文档里写 SLAVE_ID 时，顺手加了一段 BOM 说明，理由是"这俩词在同一块板子上都出现，放一起方便查"。

问题：读者正跟着通信协议走（SLAVE_ID → 寄存器 → ACK/NACK），突然跳到物料兼容性（BOM → 检测电阻 → 补偿网络），脑子得换档。独立小节会让读者以为"BOM 是协议的一部分"，但它是生产层面的事。

正确做法：BOM 缩成一句话挂在 SLAVE_ID 下面当附注——

> SLAVE_ID = 0x09H。注意同一板型可能混用 ISL88731A 和 BQ24780S（SLAVE_ID = 0x12H），外围元件不完全兼容，替换前需对照 BOM 确认检测电阻和补偿网络。

这样两个词都在，但读者不用换档——附注是从 SLAVE_ID 的电路上下文自然延伸出来的，不会造成"跑题到生产"的错觉。

## 修板子时什么时候该关心协议

修主板的人不需要懂全部协议，但在以下四个节点上，必须从电路视角切换到协议视角：

### 1. 示波器看到波形，但 EC 还是说不通

**现象：** SDA/SCL 有波形、电平正常、上拉电阻量过没问题，但 EC 读充电芯片寄存器一直超时。

**协议视角：**
- 检查 **ACK/NACK**——逻辑分析仪抓一帧，看第 9 个时钟位是不是 ACK。地址错了从设备不回 ACK，EC 拿到的就是全 FF
- 核对 **SLAVE_ID**——原理图上标的地址跟芯片 datasheet 一致吗？ISL88731A 是 0x09H，BQ24780S 是 0x12H，别混了
- 查 **I2C 超时**——SMBus 有 25-35ms 超时限制，从设备如果在这个窗口内没响应，主设备直接复位总线

### 2. 充电芯片不充电，但寄存器值看起来"对"

**现象：** EC 写 ChargeCurrent() = 4A，读出来也是 4A，芯片就是不输出电流。

**协议视角：**
- 不是所有寄存器写了就能生效——比如 BQ24780S 的 ChargeOption0() bit[0] CHRG_INHIBIT = 1 会直接禁充，哪怕 ChargeCurrent 设对了
- **寄存器映射**里有相互依赖关系——"这个值对了"不代表"这个寄存器组对了"
- 发完写命令后读回来验证——写进去了不意味着芯片内部状态机吃进去了

### 3. 换 CPU 后 VCore 电压不对

**现象：** 同一块主板换了 CPU，VCore 电压不匹配，但 VRM 芯片没坏。

**协议视角：**
- **SVID**——不同 CPU 的 VID 表不同，VRM 靠 SVID 三线（CLK/DATA/ALERT#）从 CPU 读取目标电压。示波器直接夹这三条线，看 CPU 有没有在发 SVID 命令
- **PSID**——上电前 VRM 读到 PSID 电平来判断平台类型。CPU 换了但 PSID 配置没改 → VRM 用错了参数表
- 两者是不同时间点的协议交互——PSID 在上电瞬间决定参数组，SVID 在运行中动态调电压

### 4. 莫名其妙降频/不开机，CPU 供电看着正常

**现象：** VCore 电压对、VRM 输出正常、温度也不高，但 CPU 就是降频或不开机。

**协议视角：**
- **PROCHOT**——这个信号不一定是 CPU 自己拉的。BQ24780S/ISL88731A 在检测到适配器过流时也会拉 PROCHOT，CPU 收到后强制降频
- 查 PROCHOT 来源：示波器夹 PROCHOT 引脚 + 同时看充电芯片的 IADP 电压——适配器电流是不是超了充电芯片的设定阈值？
- **SMBus Alert**——电池或充电芯片在故障时会主动拉低 ALERT# 通知 EC。如果 EC 收到 Alert 后没正确处理（比如没及时读故障寄存器），可能直接拉 PROCHOT 保护
